# 后端

> 本方案由 SSE 方案改造而来:**表设计完全不变,查询/状态/发送接口一字不改**,只把 1 个 SSE 长连接端点(`POST /stream`)换成 1 个增量拉取端点(`GET /poll`)。
> 核心复用:`userMessageStates._id` 既是「游标」也是「前端去重 key」。SSE 时代它叫 Last-Event-ID,短轮询里它就是 `lastEventId` 查询参数,语义完全一致。

## 表设计(不变)

userMessageStates 表

```js
Collection: userMessageStates
{
  // 收件箱:每个收件人 × 每条消息 = 一条记录
  _id: ObjectId,             // 同时用作 轮询游标(lastEventId)+ 前端去重 key
  userId: ObjectId,          // 收件人(投递目标)
  messageId: ObjectId,       // → MessageInfo._id
```

> 唯一变化:`POST /stream`(SSE 长连接)→ `GET /poll`(无状态增量拉取)。
> 相比 SSE 方案,后端**删掉了**:`SseTransportService`(心跳/反代缓冲/断连清理)、`messageCenter$` 热流、`handleNewMessage`、`getStreamingChannel`、全部 RxJS、上游 Change Stream 监听器、4h 僵尸连接与多副本广播处理。

## 短轮询代码大概逻辑

分层:**Controller → Manager → Service**。Controller 管 HTTP 入口与鉴权,Manager 管业务编排(不碰数据库),Service 是唯一操作数据库的层。

poll/controller.ts

```js
import { Controller, Get, Post, Body, Query } from '@nestjs/common';

import { TaskAdmin } from '@auth/core/task.guard';
import { AuthUser } from '@framework';
import { User } from '@schema/user/user.schema';
import { MessageManager } from '../manager/message.manager';

@Controller('message')
export class MessageController {
  constructor(private readonly messageManager: MessageManager) {}

  // ===== 短轮询入口(替代原 SSE /stream)=====

  // 增量拉取:前端定时调用,带上自己持有的游标 lastEventId。
  // 安全:身份用服务端解密出的 user._id,杜绝客户端伪造收件人。
  // 无 lastEventId → 首屏,拉未读做初始渲染并返回高水位游标;
  // 有 lastEventId → 轮询,只返回 { userId, _id: { $gt: cursor } } 的新投递。
  @Get('poll')
  @UseGuards(TaskAdmin)
  poll(
    @AuthUser() user: User,
    @Query('lastEventId') lastEventId?: string,
    @Query('limit') limit = 200,
  ) {
    return this.messageManager.pollMessages(user._id.toString(), lastEventId, +limit);
  }

  // 未读数(红点):前端首屏、tab 切换时拉一次;轮询过程中红点靠 poll 返回的 unreadCount 同步。
  @Get('unread-count')
  @UseGuards(TaskAdmin)
  async getUnreadCount(@AuthUser() user: User): Promise<{ count: number }> {
    const count = await this.messageManager.countUnread(user._id.toString());
    return { count };
  }

  // ===== 查询(与 SSE 方案完全一致)=====

  @Get('get-all')
  @UseGuards(TaskAdmin)
  getAll(
    @AuthUser() user: User,
    @Query('page') page = 1,
    @Query('pageSize') pageSize = 20,
  ) {
    return this.messageManager.getMessageList(user._id.toString(), { page: +page, pageSize: +pageSize });
  }

  @Get('get-detail')
  @UseGuards(TaskAdmin)
  getDetail(@AuthUser() user: User, @Query('messageId') messageId: string) {
    return this.messageManager.getMessageDetail(user._id.toString(), messageId);
  }

  // ===== 发送(与 SSE 方案完全一致)=====

  @Post('apply-submit')
  @UseGuards(TaskAdmin)
  applySubmit(@AuthUser() user: User, @Body() dto: any) {
    return this.messageManager.applySubmit(user._id.toString(), dto);
  }

  // 群发:写 1 条 MessageInfo + N 条 userMessageStates。⚠️ 高危,需更强的发送权限校验。
  // 短轮询下:状态行落库后无需触发任何推送,在线用户的下一次 poll 自然就会 $gt 命中。
  @Post('send')
  @UseGuards(TaskAdmin)
  send(@AuthUser() user: User, @Body() dto: any) {
    return this.messageManager.sendMessage(user._id.toString(), dto);
  }

  // ===== 状态操作(与 SSE 方案完全一致,一律按 token 里的 userId 作用)=====

  @Post('read')
  @UseGuards(TaskAdmin)
  read(@AuthUser() user: User, @Body() dto: { messageId: string }) {
    return this.messageManager.markRead(user._id.toString(), [dto.messageId]);
  }

  @Post('batch-read')
  @UseGuards(TaskAdmin)
  batchRead(@AuthUser() user: User, @Body() dto: { messageIds?: string[] }) {
    return this.messageManager.markRead(user._id.toString(), dto?.messageIds);
  }

  @Post('delete')
  @UseGuards(TaskAdmin)
  remove(@AuthUser() user: User, @Body() dto: { messageId: string }) {
    return this.messageManager.removeStates(user._id.toString(), [dto.messageId]);
  }

  @Post('batch-delete')
  @UseGuards(TaskAdmin)
  batchRemove(@AuthUser() user: User, @Body() dto: { messageIds: string[] }) {
    return this.messageManager.removeStates(user._id.toString(), dto.messageIds);
  }
}
```

message.manager.ts(只做业务编排,不碰数据库)

```js
import { Injectable, Logger } from '@nestjs/common';
import { MessageService } from '../service/message.service';

// 下发给前端的统一 payload:本体内容 + 当前用户的投递状态,扁平成一个对象。
// _id 用的是【状态表 userMessageStates 的 _id】,不是本体 _id ——
// 因为每个收件人对同一条消息各有一条状态记录,游标 / 去重必须按收件人维度走。
export interface MessagePayload {
  _id: string;              // = userMessageStates._id,既是轮询游标也是前端去重 key
  messageId: string;        // = MessageInfo._id
  isRead: boolean;          // 当前用户对本条的已读状态
  title: string;
  content: string;
  linkUrl: string;
  sendUserId: string;
  sendMessageType: string;
  sendMessageTime: Date;
}

// 单次拉取上限。新投递积压过多时只下发最新一批,
// 其余让前端走 GET /get-all 翻页,避免一次性灌爆响应。
const POLL_LIMIT = 200;

@Injectable()
export class MessageManager {
  private readonly logger = new Logger(MessageManager.name);

  // 只依赖 Service:Manager 不知道 mongoose / 集合的存在,所有 DB 访问都委托下去。
  constructor(private readonly messageService: MessageService) {}

  // 工具:把「状态记录 + 本体」join 成下发用的扁平 payload。纯映射,无 DB。
  private toPayload(state: any, body: any): MessagePayload {
    return {
      _id: state._id?.toString(),
      messageId: state.messageId?.toString(),
      isRead: !!state.isRead,
      title: body?.title,
      content: body?.content,
      linkUrl: body?.linkUrl,
      sendUserId: body?.sendUserId?.toString(),
      sendMessageType: body?.sendMessageType,
      sendMessageTime: body?.sendMessageTime,
    };
  }

  // 工具:把一批状态记录批量 join 本体,组装成 payload[]。
  // 编排留在 Manager,唯一那句 DB 查询(按 messageId 批量取本体)委托给 Service。
  private async joinBodies(states: any[]): Promise<MessagePayload[]> {
    if (states.length === 0) return [];
    const messageIds = states.map((s) => s.messageId);
    const bodies = await this.messageService.findBodiesByIds(messageIds); // ← DB 在 Service
    const bodyMap = new Map(bodies.map((b: any) => [b._id.toString(), b]));

    return states
      .map((s) => {
        const body = bodyMap.get(s.messageId.toString());
        return body ? this.toPayload(s, body) : null; // 本体缺失则跳过该条
      })
      .filter((p): p is MessagePayload => p !== null);
  }

  // ★ 短轮询核心:增量拉取(纯编排,查询全部走 Service)
  //   a) lastEventId 存在  → 轮询,返回「该状态ID之后」的所有新投递($gt 游标)
  //   b) lastEventId 不存在 → 首屏,拉全部 unread 做初始渲染
  //
  //   ⚠️ 字符串 lastEventId → ObjectId 的转换 + 非法兜底:转换放 Service(贴近 DB),
  //   Service 对非法 id 抛错,Manager 这里 try/catch 兜底为「返回空」,
  //   让前端保持现有游标继续轮询,而不是退化重灌历史。
  async pollMessages(
    currentUserId: string,
    lastEventId?: string,
    limit = POLL_LIMIT,
  ): Promise<{ list: MessagePayload[]; lastEventId: string | undefined; unreadCount: number; hasMore: boolean }> {
    let states: any[];

    if (lastEventId) {
      this.logger.log(`[MessageManager] 用户 ${currentUserId} 轮询,检索状态ID大于 ${lastEventId} 的新投递...`);
      try {
        states = await this.messageService.findStatesAfterCursor(currentUserId, lastEventId, limit);
      } catch {
        this.logger.warn(`[MessageManager] lastEventId 非法: ${lastEventId},本轮返回空`);
        states = [];
      }
    } else {
      this.logger.log(`[MessageManager] 用户 ${currentUserId} 首屏,拉取历史未读消息...`);
      states = await this.messageService.findUnreadStates(currentUserId, limit);
    }

    const list = await this.joinBodies(states);
    const unreadCount = await this.messageService.countUnread(currentUserId);
    const nextCursor = await this.resolveHighWaterMark(currentUserId, lastEventId, states);

    return {
      list,
      lastEventId: nextCursor,
      unreadCount,
      hasMore: states.length >= limit, // 提示前端:可能被 limit 截断,需走 get-all 翻页补齐
    };
  }

  // 计算「下一轮游标」=该用户当前最大的状态 _id(高水位)。决策在 Manager,查询在 Service。
  //   - 轮询分支:states 升序,末尾就是最新,直接取即可,省一次查询。
  //   - 首屏分支:findUnreadStates 是按未读筛的,若最新一条已被读过就不在结果里,
  //     直接拿它当游标会偏小、下轮重复拉 → 必须单独查全集最大 _id 作高水位。
  private async resolveHighWaterMark(
    currentUserId: string,
    lastEventId: string | undefined,
    states: any[],
  ): Promise<string | undefined> {
    if (lastEventId) {
      // 轮询:有新数据就推进游标,没新数据就保持原游标不动
      return states.length ? states[states.length - 1]._id.toString() : lastEventId;
    }
    // 首屏:取该用户全部状态行里最大的 _id(含已读),作为之后增量轮询的起点
    const newest = await this.messageService.findNewestState(currentUserId);
    return newest?._id?.toString();
  }

  // 未读数(红点):薄转发,DB 在 Service。
  countUnread(currentUserId: string): Promise<number> {
    return this.messageService.countUnread(currentUserId);
  }

  // ===== 查询 / 状态 / 发送:Manager 只编排,DB 一律委托 Service =====

  // 消息列表(分页):分页参数换算在 Manager,两条查询走 Service;join 复用 Manager.joinBodies。
  async getMessageList(
    currentUserId: string,
    { page = 1, pageSize = 20 }: { page?: number; pageSize?: number },
  ): Promise<{ list: MessagePayload[]; total: number; page: number; pageSize: number }> {
    const { states, total } = await this.messageService.findStatesPaged(
      currentUserId,
      (page - 1) * pageSize,
      pageSize,
    );
    return { list: await this.joinBodies(states), total, page, pageSize };
  }

  // 单条详情:越权判断(查不到即 null)在 Manager,查询走 Service。
  async getMessageDetail(currentUserId: string, messageId: string): Promise<MessagePayload | null> {
    const state = await this.messageService.findOneState(currentUserId, messageId);
    if (!state) return null;
    const [payload] = await this.joinBodies([state]);
    return payload ?? null;
  }

  // 标记已读:不传 messageIds 则一键全部已读。权限范围(只动自己)由 Service 的 query 保证。
  markRead(currentUserId: string, messageIds?: string[]): Promise<any> {
    return this.messageService.updateRead(currentUserId, messageIds);
  }

  // 删除:只删当前用户自己的状态行,本体不动。
  removeStates(currentUserId: string, messageIds: string[]): Promise<any> {
    return this.messageService.deleteStates(currentUserId, messageIds);
  }

  // 群发:Manager 组装本体字段 + 收件人名单(纯逻辑),写入交给 Service。
  // 单机 Mongo 无事务,Service 内部「先写本体再写状态行」顺序写入 + 失败补偿。
  // 短轮询下:状态行落库即可,无需任何推送 —— 在线用户下一次 poll 的 $gt 查询会自然命中。
  async sendMessage(
    sendUserId: string,
    dto: { title: string; content: string; linkUrl?: string; sendMessageType: string; receiverIds: string[] },
  ): Promise<{ messageId: string; delivered: number }> {
    const body = {
      sendUserId,
      title: dto.title,
      content: dto.content,
      linkUrl: dto.linkUrl,
      sendMessageType: dto.sendMessageType,
      sendMessageTime: new Date(),
    };
    // 注意:状态行的 messageId 由 Service 在事务内拿到本体 _id 后回填,Manager 只给收件人名单。
    const { messageId, delivered } = await this.messageService.createMessageWithStates(
      body,
      dto.receiverIds,
    );
    return { messageId, delivered };
  }

  // 提交申请:非直接发送,落审批队列(纯业务,审批集合的写入也走 Service)。
  async applySubmit(applicantId: string, dto: any): Promise<any> {
    this.logger.log(`[MessageManager] 用户 ${applicantId} 提交群发申请,待审批`);
    // TODO: this.messageService.createApply(applicantId, dto);状态 pending;审批通过后再 sendMessage(...)
    return { applied: true };
  }
}
```

message.service.ts(只做数据库操作,无业务判断)

```js
import { Injectable } from '@nestjs/common';
import { InjectModel } from '@nestjs/mongoose';
import { Model, Types } from 'mongoose';

// 这一层是唯一允许出现 Model / Types / 集合查询语句的地方。
// Manager 调用这里的方法,自己不感知 mongoose 的存在。
@Injectable()
export class MessageService {
  constructor(
    @InjectModel('userMessageStates') private readonly stateModel: Model<any>,
    @InjectModel('MessageInfo') private readonly messageModel: Model<any>,
  ) {}

  // ===== 状态表查询 =====

  // 增量:游标之后的新投递。字符串 → ObjectId 的转换 + 非法抛错都在这层(贴近 DB)。
  // { $gt: '字符串' } 在 BSON 类型比较中不命中,必须转回 ObjectId;非法 id 让 new ObjectId 抛错,
  // 由上层 Manager try/catch 兜底。升序返回,保证调用方取末尾即最新游标。
  findStatesAfterCursor(userId: string, lastEventId: string, limit: number): Promise<any[]> {
    const cursor = new Types.ObjectId(lastEventId); // 非法时抛错,交 Manager 兜底
    return this.stateModel
      .find({ userId, _id: { $gt: cursor } })
      .sort({ _id: 1 })
      .limit(limit)
      .lean();
  }

  // 首屏:未读状态最新一批(按 _id 倒序)。
  findUnreadStates(userId: string, limit: number): Promise<any[]> {
    return this.stateModel
      .find({ userId, isRead: false })
      .sort({ _id: -1 })
      .limit(limit)
      .lean();
  }

  // 高水位:该用户全部状态行里 _id 最大的一条(含已读)。
  findNewestState(userId: string): Promise<any | null> {
    return this.stateModel.findOne({ userId }).sort({ _id: -1 }).lean();
  }

  // 列表分页:返回 states + total。
  async findStatesPaged(
    userId: string,
    skip: number,
    limit: number,
  ): Promise<{ states: any[]; total: number }> {
    const [states, total] = await Promise.all([
      this.stateModel.find({ userId }).sort({ _id: -1 }).skip(skip).limit(limit).lean(),
      this.stateModel.countDocuments({ userId }),
    ]);
    return { states, total };
  }

  // 单条:按 (userId + messageId) 锁定本人的投递记录。
  findOneState(userId: string, messageId: string): Promise<any | null> {
    return this.stateModel.findOne({ userId, messageId }).lean();
  }

  // 未读数(红点):走 { userId, isRead } 索引。
  countUnread(userId: string): Promise<number> {
    return this.stateModel.countDocuments({ userId, isRead: false });
  }

  // ===== 本体表查询 =====

  // 批量 join:按 messageId 一次 $in 取本体,避免 N+1。
  findBodiesByIds(messageIds: any[]): Promise<any[]> {
    return this.messageModel.find({ _id: { $in: messageIds } }).lean();
  }

  // ===== 写操作 =====

  // 标记已读:只动 (userId + 指定 messageIds) 的未读行;不传则该用户全部未读置已读。
  // userId 始终参与 query,杜绝越权改别人状态。
  updateRead(userId: string, messageIds?: string[]): Promise<any> {
    const filter: any = { userId, isRead: false };
    if (messageIds?.length) filter.messageId = { $in: messageIds };
    return this.stateModel.updateMany(filter, { $set: { isRead: true } });
  }

  // 删除:只删当前用户自己的状态行,本体不动。
  deleteStates(userId: string, messageIds: string[]): Promise<any> {
    return this.stateModel.deleteMany({ userId, messageId: { $in: messageIds } });
  }

  // 群发写入(单机 Mongo,无事务):先写本体,再按本体 _id 写 N 条状态行。
  // ⚠️ 非原子:若 insertMany 失败,本体已落库会成为「孤儿本体」——
  //   影响可控:本体存在但无任何状态行 = 没人收到,不会错发;可由定时任务清理孤儿,
  //   或在 insertMany 失败时补偿删除刚写的本体(见下方 catch)。
  //   若后续上了副本集,可把这两步包进 session.withTransaction 即恢复原子性。
  async createMessageWithStates(
    body: any,
    receiverIds: string[],
  ): Promise<{ messageId: string; delivered: number }> {
    const created = await this.messageModel.create(body);
    const states = receiverIds.map((userId) => ({
      userId,
      messageId: created._id, // 回填本体 _id
      isRead: false,
    }));
    try {
      await this.stateModel.insertMany(states);
    } catch (e) {
      // 补偿:状态行写失败则回删本体,避免留下孤儿(尽力而为)
      await this.messageModel.deleteOne({ _id: created._id }).catch(() => void 0);
      throw e;
    }
    return { messageId: created._id.toString(), delivered: states.length };
  }
}
```

> 分层结果:**Controller**(HTTP 入口,不变)→ **Manager**(只做业务编排:首屏/增量分支、高水位决策、join 组装、越权判断,不出现任何 mongoose 语句)→ **Service**(唯一碰数据库的层:所有 `find/count/insert/update/delete`,以及字符串→ObjectId 转换、群发事务)。
> 对比 SSE 方案:**删掉**了 `SseTransportService`、`messageCenter$`、`handleNewMessage`、`getStreamingChannel` 与全部 RxJS;**新增** `MessageService` 承接 DB,Manager 由「直接注入 repo」改为「注入 Service」。`pollMessages` 的增量逻辑即原 `getHistoryMessages` 的 `$gt` 分支。

service/message.ts

```js
import request from './request'

const BASE = '/label-platform-admin/message'

// 短轮询(替代原 SSE 连接)
export function pollMessages(data?: { lastEventId?: string; limit?: number }) {
  return request.get<void, any>(`${BASE}/poll`, { params: data })
}

// ===== 以下与 SSE 方案完全一致 =====
```

useAppCore.ts

```js
import { pollMessages } from '@/service/message'

// --- 模块级状态(跨调用持久)---
let _pollTimer: ReturnType<typeof setInterval> | null = null
let _lastEventId: string | null = null
let _inFlight = false           // 防止慢请求叠加:上一轮没回来,就不发下一轮

const POLL_INTERVAL = 15_000    // 站内信 15~30s 足够,按业务对实时性的要求调

// 单次拉取。$gt 游标天然不重复投递,所以不再需要 SSE 时代的 _seenIds 去重。
const pollOnce = async () => {
  // 页面不可见(切到别的 tab / 最小化)时跳过,省请求;切回时由 visibilitychange 立即补一次
  if (_inFlight || document.hidden) return
  _inFlight = true
  try {
    const { list, lastEventId, unreadCount } = await pollMessages({
      lastEventId: _lastEventId ?? undefined,
    })

    // 推进游标:后端已算好高水位(首屏=全集最大 _id,轮询=新数据末尾或保持原值)
    if (lastEventId) _lastEventId = lastEventId

    // list 升序逐条入库;前端列表 / 红点据此更新
    for (const msg of list) messageStore.addMessage(msg)
    messageStore.setUnreadCount(unreadCount)
  } catch {
    // 网络抖动:本轮丢弃,下一个 tick 自然重试,无需指数退避
  } finally {
    _inFlight = false
  }
}

const startPolling = () => {
  if (_pollTimer) return
  pollOnce()                                    // 首屏立即拉一次,建立游标 + 渲染未读
  _pollTimer = setInterval(pollOnce, POLL_INTERVAL)
}

const stopPolling = () => {
  if (_pollTimer) clearInterval(_pollTimer)
  _pollTimer = null
}

// tab 切回前台时立即补一次,体验接近实时
const onVisibilityChange = () => {
  if (!document.hidden) pollOnce()
}
document.addEventListener('visibilitychange', onVisibilityChange)
```

> 对比 SSE 的 `useAppCore.ts`:整段 `fetchEventSource`(onopen / onmessage / onclose / onerror、AbortController、指数退避、`_seenIds` 去重)被替换成一个 `setInterval` 轮询循环,代码量更少。`connectSSE / disconnectSSE` 对应换成 `startPolling / stopPolling`,上层调用处改个名即可,生命周期挂载点(登录后启动、登出停止)不变。

# 取舍与调优

## SSE vs 短轮询 对比

| 维度 | SSE(原方案) | 短轮询(本方案) |
|---|---|---|
| 实时性 | 秒级即时 | 延迟 ≈ 轮询间隔(15~30s) |
| 多副本部署 | 要 Change Stream + 每实例监听,复杂 | **天然无状态**,任意 LB,零额外工作 |
| 长连接 / 心跳 / 反代缓冲 | 都要处理 | **全不需要** |
| 服务端资源 | 每用户 1 条常驻连接 + 订阅 | 无常驻,只有瞬时请求 |
| DB 压力 | 低(事件驱动) | 较高(空轮询也查库),靠索引 + 间隔控制 |
| 代码量 | 大 | **砍掉一大半** |
| 表设计 | — | **完全一致** |
| 业务接口 | — | **仅 stream→poll,其余不动** |

## 性能要点

- **QPS 估算** = `在线用户数 ÷ 轮询间隔`。例:5000 在线 / 15s ≈ 333 QPS,其中绝大多数是「返回空 list」的廉价 `$gt` 查询(命中复合索引,定位游标后顺序扫一小段),完全扛得住。
- **红点优化**:若不想每轮都 `countUnread`,可去掉 poll 响应里的 `unreadCount`,改由前端「收到新消息自增、点已读自减」,只在首屏 + tab 切换时调一次 `GET /unread-count`(与原文档红点策略一致)。
- **去重可省**:`$gt cursor` 严格大于,不会重复投递同一条,因此前端不再需要 `_seenIds`。保留与否取决于是否担心后端游标计算的边界,默认可删。

## 可选增强

- **自适应间隔**:用户活跃(刚操作过)用 10s,空闲一段时间退到 60s,进一步压低空轮询 QPS。
- **长轮询(long-polling)折中**:若 15s 延迟偏长又不想缩短间隔增加空查,可让 `poll` 在服务端 hold 住请求最多 ~25s,期间发现新投递就立刻返回、否则超时返回空。实时性接近 SSE,但会重新引入「服务端需感知新消息」的复杂度——**短轮询够用就不必上**。

## 注意事项

1. **首屏游标必须取全集最大 `_id`**,不能用未读列表的首条(最新消息可能已读、不在未读结果里,游标会偏小导致重复拉)。见 `resolveHighWaterMark`。
2. **`hasMore` 截断提示**:单轮新投递超过 `limit`(200)时只返回最新一批,前端应在 `hasMore=true` 时引导用户走 `GET /get-all` 翻页补齐,避免空洞。
3. **已读状态跨设备同步**:`$gt` 只捞新投递,不捞「在别处被读」的旧消息状态变化——这点与 SSE 方案一致(SSE 也只推新消息),红点靠每轮 `unreadCount` 兜底纠正。
4. **群发与轮询解耦**:`sendMessage` 写完状态行即可返回,无需任何推送动作;在线用户的下一次 poll 自然 `$gt` 命中,天然削峰(不会像 SSE 那样瞬时 fan-out 给上万连接)。
